# Answer Render Skill

## Skill 目标

本 Skill 用于将保险问答、保障责任解释、理赔判断、药品支付判断、材料清单等结果编排成前端可展示的 Markdown + JSON 混排格式。

本 Skill 重点解决：

- 用户能快速看懂结论
- 前端能识别 JSON 并渲染卡片
- 回答不要啰嗦
- 不输出无关保障责任
- 不输出虚假确定性
- notes 放在最终答案末尾

---

## 全局输出原则

1. 输出必须是 Markdown。
2. 不要输出任何标题，包括 `#`、`##`、`###`。
3. 开头必须用 1-3 句话直接回答用户当前问题。
4. 不要用“先说结论”“保障责任”“注意事项”“下一步建议”等标题。
5. 如果适合卡片展示，在开头总结之后插入 fenced json 代码块。
6. JSON 必须是合法 JSON。
7. JSON 只给前端读取，不要在正文中解释 JSON 字段。
8. 正文不要机械重复 JSON 中所有字段。
9. 最后的提醒、限制和仍需确认事项，放在答案最后。
10. 不要输出与用户当前问题无关的完整保障责任列表。
11. 不要因为检索到多个责任，就把所有责任全部输出。
12. 如果缺少关键信息，要明确说“需要确认什么”，不要假设。
13. 所有确定结论必须基于已知条款或检索内容。
14. 涉及理赔时，避免承诺性表达，应使用“可以申请”“大概率支持”“需要确认”“最终以保险公司审核为准”等表达。

---

## 回答整体结构

推荐结构：

```text
第一段：1-3 句话直接回答用户问题。

第二段：如适合，输出 fenced json 卡片。

第三段：补充解释核心原因，说明关键限制。

最后一段：列出仍需确认的信息或最终审核提醒。
```

不要加标题。

---

## JSON 输出规则

所有 JSON 必须放在 fenced code block 中：

````markdown
```json
{
  "type": "claimcard"
}
```
````

禁止输出：

```text
下面是 JSON：
```

禁止在 JSON 后解释每个字段是什么意思。

## 支持的卡片类型

### 1. policyselectcard

用于用户没有明确产品或版本，需要选择保单。

```json
{
  "type": "policyselectcard",
  "title": "请选择保险产品版本",
  "reason": "",
  "options": [
    {
      "policy_name": "",
      "policy_version": "",
      "description": ""
    }
  ]
}
```

---

### 2. claimcard

用于理赔判断、赔不赔、能不能报销。

```json
{
  "type": "claimcard",
  "claim_scene": {
    "claim_type": "",
    "coverage_name": "",
    "support_status": "支持/不支持/待确认",
    "confidence": "高/中/低",
    "reason": ""
  },
  "conditions": [
    {
      "name": "",
      "status": "已满足/未满足/待确认",
      "description": ""
    }
  ],
  "estimated_payment": {
    "claim_amount": null,
    "deductible": null,
    "pay_ratio": "",
    "estimated_result": null,
    "calculation_note": ""
  },
  "missing_info": [],
  "notes": []
}
```

如果没有金额测算，省略 `estimated_payment`。

如果没有缺失信息，`missing_info` 可以为空数组。

`notes` 可以为空数组，但真正给用户看的注意事项应放在答案最后，不要只藏在 JSON 里。

---

### 3. coveragecard

用于保障责任解释。

```json
{
  "type": "coveragecard",
  "coverage": {
    "name": "",
    "summary": "",
    "insured_amount": "",
    "deductible": "",
    "pay_ratio": "",
    "hospital_scope": "",
    "key_limits": []
  }
}
```

---

### 4. materialcard

用于理赔材料清单。

```json
{
  "type": "materialcard",
  "claim_type": "",
  "materials": [
    {
      "name": "",
      "required_level": "必须提供/可能需要/有则提供",
      "description": "",
      "how_to_get": "",
      "normalized_name": ""
    }
  ]
}
```

## 不同场景的输出要求

### 场景一：产品版本不明确

用户问：

“我买了沪惠保，住院自费能不能赔？”

如果问题涉及版本差异，应回答：

```markdown

```json
{
  "type": "policyselectcard",
  "title": "请选择保险产品版本",
  "reason": "用户只提到沪惠保，但住院理赔规则在不同版本中存在差异。",
  "options": [
    {
      "policy_name": "2025版沪惠保",
      "policy_version": "普通版",
      "description": "适用于参加上海市基本医疗保险的在保人员"
    },
    {
      "policy_name": "2025版沪惠保",
      "policy_version": "关爱版",
      "description": "适用于参加上海市市民社区医疗互助帮困计划的人员"
    },
    {
      "policy_name": "2025版沪惠保",
      "policy_version": "新市民版",
      "description": "适用于在上海部分大型企业工作并参加当地基本医保的务工人员"
    }
  ]
}
```

你选定版本后，我可以继续帮你判断能不能赔、赔付比例和需要补充的信息。
```

注意：实际输出时，fenced json 必须完整闭合。

    "coverage_name": "特定住院自费医疗费用保险金",
    "support_status": "支持",
    "confidence": "中",
    "reason": "用户明确为新市民版且不是既往症；该版本允许未经当地医保结算申请，但赔付比例降低。"
  },
  "conditions": [
    {
      "name": "产品版本",
      "status": "已满足",
      "description": "用户明确为2025版沪惠保新市民版。"
    },
    {
      "name": "医保结算",
      "status": "未满足",
      "description": "未经过当地基本医保结算，因此适用降低后的赔付比例。"
    },
    {
      "name": "既往症状态",
      "status": "已满足",
      "description": "用户说明不是既往症，适用非既往症20%的赔付比例。"
    },
    {
      "name": "费用范围",
      "status": "待确认",
      "description": "需确认费用是否属于条款约定的特定住院自费医疗费用。"
    }
  ],
  "missing_info": [
    "住院医院是否为上海市二级及以上医保定点医院普通住院部",
    "费用是否属于特定住院自费医疗费用",
    "是否超过年度免赔额"
  ],
  "notes": []
}
```

还需要确认住院医院、费用明细和免赔额情况。最终赔付金额以保险公司审核为准。
```

  "conditions": [

      "name": "产品版本",
      "status": "已满足",
      "description": "用户明确为2025版沪惠保新市民版。"
    },
    {
      "name": "医保结算",
      "status": "未满足",
      "description": "未经过当地医保结算，因此适用降低后的赔付比例。"

    {
      "name": "既往症状态",

### 场景四：明确不支持

用户问：

“医生一次给我开了两个月特药处方，沪惠保能全部赔吗？”

应回答：

```markdown
不能直接按两个月全部赔。条款对国内特定高额药品和海外特殊药品有明确限制：每次药品处方超过一个月以上部分的药品费用，通常不承担给付责任。

```json
{
  "type": "claimcard",
  "claim_scene": {
    "claim_type": "特定高额药品费用理赔",
    "coverage_name": "国内特定高额药品费用保险金/海外特殊药品费用保险金",
    "support_status": "不支持",
    "confidence": "高",
    "reason": "每次药品处方超过一个月以上部分的药品费用属于责任免除范围。"
  },
  "conditions": [
    {
      "name": "处方时长",
      "status": "未满足",
      "description": "用户描述一次开具两个月处方，超过一个月以上部分存在免责风险。"
    }
  ],
  "missing_info": [],
  "notes": []
}
```

一个月以内部分是否能赔，还要继续看药品是否在目录、适应症是否符合、处方医生和购药渠道是否满足要求。
```

## 语言风格要求

回答应该像专业但好懂的保险客服：

- 直接
- 克制
- 不吓人
- 不堆条款
- 不过度承诺
- 不使用复杂法律话术
- 必要时提醒“最终以保险公司审核为准”

---

## 禁止输出

禁止以下表达：

- “一定可以赔”
- “肯定能报”
- “100% 能赔”
- “不用担心，肯定可以”
- “我保证”
- “保险公司一定会赔”
- “根据常识判断”
- “应该差不多”
- “条款没说但我认为”

除非条款明确且所有条件都满足，否则不能输出确定承诺。

---

## notes 规则

`notes` 字段可以保留为空数组。

真正面向用户的注意事项应写在答案最后，例如：

- “最终赔付金额以保险公司审核为准。”
- “还需要确认费用是否属于条款约定范围。”
- “如果药品已获医保报销，该产品可能不再受理对应药品费用理赔申请。”
- “如果处方超过一个月，超过部分可能不承担责任。”

不要把关键提醒只放在 JSON 里。
