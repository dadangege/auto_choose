# 2026-05-25 晚间进展记录

## 本晚目标

把保险问答从“单次 LLM 生成 Markdown”升级成可测试、可观察、可展示的 skill 编排原型：

- 用户可以随便问保障、药品、医院、投保、理赔等混合问题。
- 系统先本地拆分 intent，再规划每个 intent 应该输出哪些卡片。
- LLM 输出不再是一个整体答案后面跟一个大 JSON，而是保持“内容 + JSON + 内容 + JSON”的相邻结构。
- 前端可以把 fenced JSON 渲染成业务卡片，并能观察每一步耗时。

## 已完成能力

### 1. Skill 编排结构

`insurance-qa/SKILL.md` 已从普通知识提示词升级为顶层编排器，明确要求：

- 先做产品版本选择。
- 再拆分多 intent。
- 每个 intent 单独判断。
- 每个 intent 都有自己的主卡、辅助卡和证据锚点。
- 多 intent 输出必须保持相邻的“内容 + JSON”结构。

新增参考文件：

- `references/intent-routing.md`
- `references/card-taxonomy.md`
- `references/source-grounding.md`
- `references/answer-composition.md`

这些文件借鉴了 `nature-skills` 的思路：总控 skill 负责编排，细节规则拆到 references，输出是 artifact bundle，而不是单纯聊天文本。

### 2. 本地 cardPlan

新增 `orchestrator/card-planner.js`。

现在本地编排器会为每个 intent 生成：

- `primary_card`
- `supporting_cards`
- `answer_job`
- `source_anchors`

示例问题：

```text
我买的是新市民版，住院自费3万，没走当地医保，我不是既往症。另外奥希替尼能不能报？
```

会拆成：

```text
hospital_self_pay
  primary: claimcard
  support: evidencecard, nextstepcard
  anchors: R001, R002

domestic_drug
  primary: claimcard
  support: drugcheckcard, evidencecard, nextstepcard
  anchors: D001, D002
```

### 3. 两段式 LLM 编排

新增 `两段式编排` Prompt 模式。

实际会调用两次 LLM：

```text
用户问题
-> 本地 router / slot / rule / cardPlan
-> 第 1 次 LLM：Planner，输出 answer_plan
-> 第 2 次 LLM：Renderer，基于 answer_plan + wiki 输出最终回答和 JSON
-> 前端解析 JSON 并渲染卡片
```

第一段负责生成结构化草稿：

- 问题概括
- 全局上下文
- sections
- 每个 section 的回答方向
- 必须覆盖的事实
- 卡片顺序
- 缺失字段
- 需要的 wiki 域

第二段负责最终展示：

- 用户可读正文
- fenced JSON 卡片
- 多 intent 相邻输出
- 待确认事项
- 证据锚点

### 4. 快速 Wiki 模式

保留并强化了快速 Wiki 模式。

本地会先判断需要哪些 wiki 文件，只把命中的片段发给 LLM，而不是把全部 raw 文件或全部 skill 都塞进上下文。

这能降低 prompt 体量，提高响应速度。

### 5. 前端测试台

测试页面新增和优化：

- Prompt 模式：快速 Wiki、完整 Skill、解读报告、两段式编排。
- 运行轨迹：产品识别、意图拆分、LLM 判断、卡片渲染。
- 能力地图：解释当前测试台能测哪些能力。
- 本地编排计划：展示每个 intent 的主卡、辅助卡、证据锚点。
- 两段式输出面板：第一段 answer_plan 和第二段最终回答分开展示。
- 耗时统计：Prompt 体量、首字、第 1 段、第 2 段、总耗时、输出长度。
- JSON 卡片渲染：理赔卡、保障卡、药品核验卡、证据链卡、下一步卡、版本对比卡、免责卡等。

### 6. 产品解读样例

新增更接近产品展示的“产品解读”视图，用来验证卡片和报告型 UI 的表达效果。

它不是实时调用 LLM，而是一个静态展示样例，方便观察：

- 保障责任矩阵
- 版本入口
- 新市民版住院计算
- 奥希替尼证据链
- JSON 卡片视觉样式

## 当前限制

1. 第一段 Planner 目前不是流式输出。
   - 前端显示的是第一段完整响应耗时，不是真正的第一段首 token。

2. 第一段结构还主要依靠 prompt 约束。
   - 后续应加入 JSON parse、schema 校验和自动修复。

3. 两段式会多问一次 LLM。
   - 效果更稳，但总耗时会增加。

4. 视觉样式主要由前端 CSS 控制。
   - skill 负责语义结构和卡片类型，字号、颜色、间距仍由 `style.css` 控制。

## 下一步建议

### 优先级 1：本地 deterministic planner

把第一段 `answer_plan` 尽量改成本地生成：

```text
router + cardPlan -> local answer_plan -> 只调用一次 LLM renderer
```

好处：

- 速度更快。
- 结构更稳定。
- 成本更低。

### 优先级 2：Planner JSON 校验

给第一段增加：

- JSON.parse
- schema validation
- repair prompt
- 本地兜底 plan

### 优先级 3：视觉渲染规则文档

新增类似 `references/visual-render.md` 的文档，定义：

- 哪些状态用哪些颜色。
- 哪些卡片用强调样式。
- 多 intent 页面如何分区。
- 待确认、支持、不支持如何展示。

这部分应该由前端消费语义字段，而不是让 LLM 直接输出字号和颜色。

## 当前结论

这版已经从“测试几个 skill 能不能回答”升级成了一个可真实演示的保险 QA 编排原型：

- 本地负责拆解和规则判断。
- wiki 负责降低上下文体量。
- LLM 负责表达和补齐。
- JSON 卡片负责前端展示。
- 两段式模式负责验证“先规划、再渲染”的产品思路。
